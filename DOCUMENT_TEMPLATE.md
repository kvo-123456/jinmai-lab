## 📝 文档风格指南

### 1. 标题层级

- 使用 `#` 到 `######` 表示不同层级的标题
- 避免跳过标题层级
- 标题应简洁明了，概括章节内容

### 2. 代码样式

- 使用反引号 `` ` `` 包裹单行代码
- 使用三个反引号 `` ``` `` 包裹多行代码块
- 为代码块添加语言标识，如 `` ```bash ``、`` ```javascript `` 等
- 代码块应包含必要的注释

### 3. 列表样式

- 使用数字编号列表表示步骤或序列
- 使用无序列表表示并列内容
- 列表项应简洁明了

### 4. 表格样式

- 使用 Markdown 表格语法
- 表头与内容之间使用 `|-----|-----|` 分隔
- 表格应包含表头

### 5. 链接样式

- 使用 `[链接文本](链接地址)` 格式
- 链接文本应简洁明了，描述链接内容
- 避免使用 "点击这里" 等模糊的链接文本

### 6. 图片样式

- 使用 `![图片描述](图片地址)` 格式
- 图片描述应简洁明了
- 考虑使用响应式图片，添加宽度限制

### 7. 强调样式

- 使用 `*斜体*` 表示强调
- 使用 `**粗体**` 表示重要内容
- 使用 `***粗斜体***` 表示特别重要的内容

### 8. 注意事项

- 使用 `> **注意**：` 表示注意事项
- 使用 `> **警告**：` 表示警告信息
- 使用 `> **提示**：` 表示提示信息

### 9. 文档结构

- 每个文档应包含清晰的目录
- 内容应按照逻辑顺序组织
- 每个章节应有明确的主题
- 文档应包含常见问题和总结

### 10. 语言风格

- 使用简洁明了的语言
- 避免使用过于专业的术语
- 首次使用专业术语时应解释
- 使用主动语态
- 保持一致的术语和命名

## 🚀 文档维护指南

### 1. 文档更新流程

1. 当代码或功能发生变更时，同步更新相关文档
2. 更新文档时，在文档末尾添加更新日志
3. 确保文档与代码版本一致

### 2. 更新日志格式

```
## 更新日志

- **2024-01-01**：初始版本
- **2024-02-15**：添加新功能说明
- **2024-03-30**：修复文档错误
```

### 3. 文档审查

- 定期审查文档，确保内容准确无误
- 邀请其他团队成员审查文档
- 根据审查反馈更新文档

### 4. 文档归档

- 当文档版本过时，应归档或更新
- 归档文档应清晰标记为过时
- 提供指向最新文档的链接

---
